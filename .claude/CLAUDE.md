# Claude Instructions

## Style

- Be concise. Lead with the answer, not the reasoning.
- No emojis unless asked.
- No trailing summaries — the diff speaks for itself.

## Code

- Prefer editing existing files over creating new ones.
- Don't add comments, docstrings, or type annotations to code you didn't change.
- Don't add error handling for scenarios that can't happen.
- Keep solutions simple — minimum complexity for the task at hand.

## Workflow

- Read files before editing them.
- Ask before taking irreversible actions (deleting files, force pushing, etc.).
- Don't commit unless explicitly asked.
- Always write a failing test before writing implementation code (TDD).
